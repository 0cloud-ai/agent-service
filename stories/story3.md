# 房间里的房间

## 一

又过了两个月。

agent-service 在公司里铺开了。不只是支付网关团队在用——用户中心、通知服务、数据平台，每个团队都在服务器上有自己的目录，目录里住着各自的工程会话和服务面板。老张的那棵目录树越来越茂盛，从最初的三五个项目变成了二十几个。

但一个新问题浮出了水面。

周三下午的周会上，CTO 李昀在白板上画了一张图：

"现在每个团队都有自己的 agent-service 目录，挺好。但团队之间的协作还是老样子——开会、拉群、发文档。上周那个'会员日大促'的需求，产品提了之后，我们花了两天开了四个会，才把设计、前端、后端、支付四个团队对齐。每个团队单独干活的速度已经很快了，但对齐的成本一点没降。"

她在白板上画了四个方框，代表四个团队的目录。然后在方框之间画了一堆箭头。

"我想做一件事：让这些目录之间能直接对话。一个团队可以把另一个团队当同事来 @——不用开会，不用拉群，需求直接流进去，结果直接流回来。"

## 二

老张花了三周实现了这个功能。

核心思路很简单：每个项目目录已经有了服务面板，对外暴露了一个对话接口。那么，一个目录里的 Agent 完全可以通过这个接口去"请求"另一个目录的服务——就像陈霜在服务面板里跟支付网关团队聊天一样，只不过这次发起请求的不是人，而是另一个目录里的 Agent。

用老张的话说："每个目录对外看起来就是一个 Agent。Agent 能跟人聊天，当然也能跟另一个 Agent 聊天。"

## 三

第一个真正的考验来得很快——国庆会员日大促。

产品总监赵琳在「产品」目录的服务面板里输入了一条需求：

> 会员日大促方案：10 月 1 日到 7 日，全场会员享 8 折优惠，订单满 200 减 30，支付成功后发放下次可用的 15 元优惠券。需要设计出活动页面，前端实现页面和交互，后端实现折扣和满减逻辑，支付团队对接优惠券发放。一周内上线。

这条消息进入了「产品」目录的工程面板。产品团队的 Agent 读完需求后，没有把它当作一个自己能独立完成的任务——它识别出这个需求横跨了四个团队。

Agent 开始拆解：

> 这个需求涉及四个团队，我来分别对接：
>
> 1. @设计团队 — 活动页面的视觉设计
> 2. @前端团队 — 页面开发和交互实现
> 3. @支付网关 — 优惠券发放逻辑
> 4. @后端团队 — 折扣和满减的核心计算

四条请求几乎同时发出，分别流入了四个目录的服务面板。

## 四

在「设计」目录的工程面板里，设计师周晴看到了一条新会话："会员日大促活动页视觉设计——来自产品目录"。

她点进去，看到的不只是一句干巴巴的需求——产品目录的 Agent 已经把上下文整理好了：活动时间、折扣规则、目标用户群、过往大促的设计风格参考。周晴不需要回去翻飞书找 PRD，所有她需要的信息都在这条会话里。

她和自己目录里的 Agent 开始协作。Agent 根据过往的设计系统生成了三版初稿，周晴选了第二版，调整了配色和排版，两个小时后把定稿推了回去。

在「设计」目录的服务面板上，产品目录的 Agent 收到了回复：

> 设计完成。活动页采用暖色系主视觉，突出"8 折"和"满 200 减 30"两个利益点。定稿附件已上传。

这条回复自动流回了「产品」目录的工程面板。产品团队的 Agent 记录下来：设计已完成。然后把设计稿转发给了「前端」目录。

## 五

与此同时，「支付网关」目录里，林远舟看到了一条熟悉的来路。

"会员日大促优惠券发放——来自产品目录"。他点进去，看到了需求：支付成功后发放 15 元优惠券，限下次使用。

他笑了一下。两个月前陈霜让他做过一模一样的事情，当时的会话还在目录里挂着。他跟 Agent 说：

> 这个我们上次做过。去看一下"支付成功后发放优惠券"那条会话，逻辑基本一样，只是优惠券面额改成 15 元，使用限制改成"下次订单可用"。

Agent 打开了那条历史会话，读完了上次的实现方案，然后生成了差异化的修改：只需要改两个参数和一个校验条件。十五分钟后，代码改完，测试通过。

结果流回「产品」目录：

> 支付网关已完成优惠券发放逻辑调整，复用了上次的实现方案，仅修改面额和使用限制参数。单元测试通过。

## 六

到了周五下午，赵琳打开「产品」目录的工程面板，看到了一张完整的进度图：

```
会员日大促
├── 设计团队    ✅ 已完成 — 活动页视觉定稿
├── 前端团队    ✅ 已完成 — 页面开发、联调通过
├── 支付网关    ✅ 已完成 — 优惠券发放逻辑
└── 后端团队    ✅ 已完成 — 折扣和满减核心逻辑
```

四个团队，一周之内，没有开过一场跨团队会议。

不是因为不需要沟通——沟通量其实比以前更大了。赵琳粗略数了一下，这条主会话下面关联的子会话超过了四十条，涉及的消息总数将近三百条。设计师和前端之间反复对了三轮交互细节，后端和支付网关讨论了满减和优惠券的叠加边界情况，产品 Agent 在中间帮忙传递上下文、同步进度、在需求有歧义的时候主动回来找赵琳确认。

但这些沟通全部发生在目录树里。每一条对话都有记录，每一个决策都可追溯。没有人需要会后写会议纪要，因为会话本身就是纪要。没有人需要问"这个需求最新的版本是什么"，因为答案就挂在目录树上，实时更新。

## 七

周五下班前，CTO 李昀在 `agent.team.dev` 上打开了根目录。

那棵目录树比两个月前又大了一圈。她展开了几个节点，粗略扫了一下各团队本周的活动：

```
▼ home                                        187
  ▼ product                                    23
    ▶ membership-day-promo                     41
    ▶ q4-roadmap                                8
  ▶ design                                     19
  ▶ frontend                                   34
  ▼ linyuanzhou                                28
    ▶ payment-gateway                          16
    ▶ coupon-service                            7
  ▶ wangxiao                                   22
  ▶ backend                                    31
  ▶ data-platform                              14
```

一百八十七条会话。人和 Agent 的对话混在一起，团队内部的工作和团队之间的协作混在一起。但它们都挂在同一棵树上，按目录组织，按时间排序，点进去就能看到完整的来龙去脉。

她想起三周前那次周会上自己画的那张图——四个方框，一堆箭头。现在那些箭头不再是白板上的虚线了。它们变成了目录之间真实流动的请求和响应，每一条都有记录，每一条都可以追溯。

李昀关掉了浏览器。

她想，这已经不是一个工具了。这是一个空间——人和 AI 住在同一个房间里，房间和房间之间可以互相喊话。工程师在里面写代码，设计师在里面出图，产品经理在里面提需求，而 Agent 在所有这些人之间穿针引线，搬运上下文，推进执行。

每个房间从外面看，就是一个聪明的同事。你不需要知道里面有几个人、有几个 AI、用的什么技术栈。你只需要 @它，说清楚你要什么，它就会给你结果。

而如果你走进去，你会看到——里面还有更多的房间。

---

*人和 AI 不该在彼此隔绝的窗口里工作。它们应该住在同一个房间。*
